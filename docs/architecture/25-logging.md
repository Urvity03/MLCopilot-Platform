# 25 — Logging Strategy

## Stack

`structlog` everywhere (API, workers, agents), JSON output in containers, pretty console in local dev (`LOG_FORMAT=console|json`).

## Standard fields

Every log line carries: `timestamp`, `level`, `event`, `logger`, `request_id`, `user_id` (when authed), `project_id` (when scoped). Request ID is generated by middleware (`X-Request-ID` honored if supplied), returned in responses and the error envelope, and **propagated into Celery task headers** so a user action traces across process boundaries.

## What gets logged

| Layer | Events |
|---|---|
| HTTP middleware | request start/end, method, path, status, duration_ms |
| Services | domain-significant transitions (experiment completed, investigation started) |
| Event bus | event appended, outbox dispatched, consumer success/failure with event id |
| Workers | task start/end/retry with task id + natural keys |
| Agents | per-node: `agent`, `node`, `latency_ms`, `tokens_in/out`, `provider`, `model`, tool calls with tool name + duration (never full prompts at INFO; DEBUG gated) |
| Providers | failovers, rate limits, invalid responses |

## Rules

1. Never log secrets, passwords, tokens, API keys, or full LLM prompts/responses at INFO+ — a structlog processor scrubs known keys defensively.
2. Errors log once, at the boundary that handles them, with `exc_info` — no double logging up the stack.
3. `LOG_LEVEL` env-controlled per service; DEBUG enables prompt/response logging for agent debugging (explicitly documented as unsafe for production).
